# Open Questions

## Q1. 发布形态

Status: narrowed on 2026-06-19

- 用户倾向：新产品应复用三个参考源的现有代码/资料能力，而不是从零重写。
- 当前保留问题：复用是“直接并入核心仓库”，还是“核心 Skill + 外部 adapter/参考依赖”。

- 是发布为单个 Codex Skill，还是 Skill + 可选外部 MCP/CLI 适配包？
- 当前判断：后者更稳，更适合“各种 AI 使用”。

## Q2. 外部执行边界

- Seedance 2.0 是否要求正式接 Dreamina CLI，还是先支持“生成任务规格，不直接执行”？
- 当前判断已更新：MVP 不能只停在 `plan-only`。至少要支持“生成审核包 -> 用户批准 -> Seedance 执行”。

## Q3. 抖音适配器边界

- 是否允许把 Douyin 自动发布作为单独可选安装项，而不是内嵌到 Skill 代码里？
- 当前判断：应该允许，否则许可证和平台漂移风险过高。

## Q4. 目标用户

- 主要面向“会配置环境的 AI 工具用户”，还是“拿来即用”的普通内容创作者？
- 当前判断：先服务前者，才能把自动化链路做稳。

## Q5. 产物格式

- 中间产物优先用 Markdown，还是 Markdown + JSON 双轨？
- 当前判断：双轨更利于通用 AI 消费与自动执行。

## Q6. 审核状态契约

- 用户审核通过这一关，是只写进 `run-ledger.json`，还是单独定义 `review-decision.json`？
- 当前倾向：单独定义。否则“待审核 / 已批准 / 退回修改”无法成为稳定接口。

## Q7. 长视频组装责任归属

- 长视频拼接应放在核心脚本内，还是放在独立适配器里？
- 当前倾向：放在核心脚本内。因为它属于稳定的本地后处理，不属于平台特有逻辑。
